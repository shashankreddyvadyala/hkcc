Heat-Kernel Context Compression

Keep what's distinctive.
Re-express the rest.

A training-free way to shrink LLM context. Put your tokens on a graph, run the heat equation, and fold redundant tokens into a few weighted centroids — sending far fewer tokens while the model sees almost the same thing.

Why it works

Two ideas do the heavy lifting

The residual is a saliency score

Diffusion pulls every token toward the average of its neighbours. A token surrounded by near-copies is already at that average, so it hardly moves — redundant. A one-of-a-kind token gets dragged a long way — distinctive. The distance it travels is the score.

Merge, don't prune

A repetitive region carries real attention mass as a group. Deleting it loses that mass and accuracy falls apart. Re-expressing it as one weighted centroid keeps it — which is why this one design choice is the whole game.

What you get

Three guarantees, for free

Linear time

Diffusion runs as a Chebyshev approximation — never the matrix exponential — so cost scales with the number of tokens, not its cube.

Nothing invented

Every representative is a weighted average of real tokens, so a compressed token can never take a value the context didn't contain.

Set a floor

Name the fidelity you'll accept and HKCC keeps the fewest tokens that stay above it — no guessing a budget.

Status

This is research-stage, and that's stated plainly. The numbers above come from synthetic data; they show the mechanism works but are not yet a real-LLM result.

The honest next step — and the thing that would make HKCC production-credible — is wiring it into a real retrieval-QA pipeline and reporting answer quality against measured token reduction. A ready-to-run harness ships in the repo: examples/rag_validation.py.

If you run it on real data, open an issue with your numbers. Real results are what this project needs most.
